요청: **$ARGUMENTS**

---

## 실행 파이프라인

> 4단계 에이전트 체인을 순서대로 실행한다.
> 각 단계의 결과에 따라 다음 단계로 진행하거나 이전 단계로 회귀한다.

---

### Phase 1. 구현 계획서 작성

계획서 저장 경로를 먼저 준비한다:

```bash
mkdir -p .claude/plans
```

**@feature-planner** 에이전트를 실행한다:

```
구현 요청: $ARGUMENTS

아래 절차에 따라 구현 계획서를 작성하세요:
1. 요구사항을 분석하고 관련 도메인과 기존 코드를 탐색하세요.
2. 사용자와 엣지 케이스를 논의하세요 (AskUserQuestion 사용).
3. 계획서를 .claude/plans/{기능명}.md 파일로 작성하세요.

계획서 작성이 완료되면 파일 경로를 보고하세요.
```

**게이트**: 계획서 파일이 생성되었는지 확인한다. 사용자에게 계획서를 보여주고 승인을 받는다.

---

### Phase 2. 구현

**@feature-implementer** 에이전트를 실행한다:

```
아래 구현 계획서를 읽고 코드를 구현하세요:
- 계획서 경로: .claude/plans/{기능명}.md

구현 순서 섹션의 체크리스트를 순서대로 실행하고,
완료 후 생성/수정된 파일 목록과 컴파일 결과를 보고하세요.
```

**게이트**: `./gradlew compileKotlin` 성공 확인.

---

### Phase 3. 테스트

**@test-runner** 에이전트를 실행한다:

```
아래 구현 계획서의 테스트 계획에 따라 테스트를 작성하고 실행하세요:
- 계획서 경로: .claude/plans/{기능명}.md

테스트 결과를 PASS/FAIL로 보고하세요.
실패 시 계획서에 엣지 케이스를 추가하고 수정 필요 항목을 명시하세요.
```

**분기**:
- **PASS** → Phase 4로 진행
- **FAIL** → 계획서가 업데이트되었으므로 **Phase 2로 회귀** (최대 3회)

---

### Phase 4. 코드 검증

**@code-validator** 에이전트를 실행한다:

```
아래 구현 계획서와 변경된 코드를 검증하세요:
- 계획서 경로: .claude/plans/{기능명}.md

컨벤션 준수, 사이드 이펙트, 불필요한 변경을 검사하고 결과를 보고하세요.
```

**분기**:
- **PASS** → 완료. 사용자에게 최종 보고.
- **FAIL** → 계획서가 업데이트되었으므로 **Phase 2로 회귀** (최대 2회)

---

## 회귀 제한

| 트리거 | 최대 회귀 횟수 |
|---|---|
| 테스트 실패 → Phase 2 | 3회 |
| 검증 실패 → Phase 2 | 2회 |
| 총 합계 | 5회 초과 시 사용자에게 판단 요청 |

회귀 시 매번 계획서의 업데이트된 내용을 implementer에게 전달한다.

---

## 최종 보고 형식

모든 Phase가 통과하면:

```
## 구현 완료 보고

### 기능: {기능명}

### 생성된 파일
- path/to/File1.kt — 역할 설명
- ...

### 수정된 파일
- path/to/File2.kt — 변경 내용
- ...

### 테스트 결과
- 단위 테스트: N개 통과
- 통합 테스트: N개 통과

### 검증 결과
- 컨벤션: PASS
- 사이드 이펙트: 없음

### 회귀 횟수
- 테스트 실패 회귀: N회
- 검증 실패 회귀: N회
```
