# Notion Page Review

노션 페이지의 하위 구조를 분석하여, 정리가 필요한 부분을 진단하고 리포트하는 스킬. 실제로 페이지를 이동하거나 생성하지 않으며, 현재 상태를 분석하고 개선안을 제안하는 데 집중한다.

## 전제 조건

- Notion MCP 연결이 활성화되어 있어야 한다.
- 사용 도구: `notion-fetch`, `notion-search` (읽기 전용, 변경 없음)

---

## 워크플로우

### Step 1: 대상 페이지 확인

사용자에게 검토할 상위 페이지를 확인한다.

- 사용자가 URL/이름을 제공한 경우 → 바로 진행
- 제공하지 않은 경우 → `notion-search`로 후보를 찾아 확인

### Step 2: 전체 구조 스캔

`notion-fetch`로 대상 페이지의 전체 구조를 파악한다.

수집할 정보:
- 전체 하위 페이지 목록 (제목, ID)
- 현재 깊이 구조 (몇 단계까지 중첩되어 있는지)
- 기존 카테고리 폴더가 있는지
- 데이터베이스 유무
- 각 하위 페이지의 하위 페이지 유무 (재귀 탐색)

### Step 3: 문제점 진단

아래 기준으로 현재 구조의 문제점을 분석한다.

#### 진단 항목

1. **미분류 페이지 (Uncategorized)**
   - 카테고리 폴더 없이 상위 페이지 바로 아래에 나열된 페이지들
   - 같은 주제인데 흩어져 있는 페이지들

2. **카테고리 불균형 (Imbalanced)**
   - 한 카테고리에 너무 많은 페이지 (10개 이상 → 세분화 필요)
   - 한 카테고리에 1개만 있는 페이지 (합칠 수 있는지 검토)

3. **깊이 문제 (Depth Issues)**
   - 모든 페이지가 1단계에 평탄하게 나열 → 카테고리화 필요
   - 불필요하게 깊은 구조 (4단계 이상) → 단순화 고려

4. **분류 오류 가능성 (Misplaced)**
   - 카테고리 이름과 맞지 않는 위치에 있는 페이지
   - 예: "AWS" 페이지가 "외국어" 카테고리 안에 있는 경우

5. **빈 카테고리 (Empty)**
   - 하위 페이지가 없는 카테고리 폴더

6. **중복 가능성 (Duplicates)**
   - 비슷한 제목의 페이지가 여러 개 존재

### Step 4: 진단 리포트 출력

분석 결과를 깔끔한 리포트로 제시한다.

```
📋 노션 페이지 구조 진단 리포트

📍 대상: 개인공부
📊 현재 상태: 하위 페이지 12개, 카테고리 0개, 깊이 1단계

---

🔴 주요 이슈

1. 미분류 페이지 12개
   모든 페이지가 카테고리 없이 최상위에 나열되어 있습니다.
   → DDD, MySQL, JPA 사용관련, LLM, coroutine, 차량 제어 시스템,
     AWS, 일본어, 영어 등

2. 관련 주제 그룹화 가능
   - 백엔드 관련: DDD, MySQL, JPA 사용관련 (3개)
   - 외국어 관련: 일본어, 영어 (2개)

---

🟡 개선 제안

1. 6개 카테고리로 분류하면 구조가 깔끔해집니다
2. 백엔드 카테고리는 ORM/DB로 세분화하면 3단계 구조 가능
3. 향후 페이지 추가 시에도 적절한 카테고리에 바로 넣을 수 있습니다

---

💡 정리를 실행하려면 "노션 정리해줘"라고 말씀해주세요.
```

### Step 5: 후속 액션 안내

리포트 마지막에 사용자가 취할 수 있는 액션을 안내한다:

- "노션 정리해줘" → `notion-page-organizer` 스킬로 실제 정리 실행
- 특정 부분만 수정하고 싶은 경우 → 어떤 부분을 변경할지 구체적으로 요청하도록 안내

---

## 진단 심각도 기준

| 심각도 | 조건 | 표시 |
|---|---|---|
| 🔴 높음 | 미분류 10개 이상 / 분류 오류 발견 | 즉시 정리 권장 |
| 🟡 보통 | 미분류 3~9개 / 카테고리 불균형 | 정리하면 좋음 |
| 🟢 낮음 | 미분류 1~2개 / 구조 양호 | 현재 상태 괜찮음 |

---

## 주의사항

- 이 스킬은 **읽기 전용**이다. 페이지를 이동하거나 생성하지 않는다.
- 진단 결과는 제안일 뿐, 사용자의 판단을 존중한다.
- 개인적인 정리 방식이 있을 수 있으므로, 리포트 후 사용자의 의견을 먼저 듣는다.
