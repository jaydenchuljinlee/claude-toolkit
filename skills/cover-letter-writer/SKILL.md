# 개발자 자기소개서 작성 스킬

채용 공고, 이력서/경력기술서를 분석하고 기업의 최신 동향을 리서치하여
사람이 직접 쓴 것 같은 자연스러운 개발자 자기소개서를 생성하는 스킬이다.

## 파이프라인

아래 단계를 순서대로 진행한다. 각 단계의 결과를 다음 단계의 입력으로 활용한다.

### 1단계: 입력 수집 및 분석

사용자로부터 다음 세 가지를 확보한다.

- **채용 공고**: 담당 업무, 필수 역량, 우대사항, 팀 문화 등을 추출한다
- **이력서/경력기술서**: 경력 사항, 프로젝트별 기술 스택과 성과, 핵심 역량을 추출한다
- **자소서 항목**: 각 질문의 의도와 글자 수 제한을 파악한다

하나라도 빠져 있으면 사용자에게 요청한다. 세 가지가 모두 있어야 다음 단계로 넘어간다.

### 2단계: 채용 공고 ↔ 경력 매핑

채용 공고의 요구사항과 경력기술서의 경험을 연결한다.

- 필수 역량에 대응하는 프로젝트/성과를 찾는다
- 우대사항에 해당하는 경험이 있으면 별도로 표시한다
- 매칭되지 않는 요구사항이 있으면, 관련된 유사 경험이나 학습 의지로 보완할 수 있는지 확인한다

이 매핑 결과는 내부적으로만 활용하며, 사용자에게 별도로 보여줄 필요는 없다.

### 3단계: 기업 리서치

web_search를 사용하여 해당 기업의 최근 동향을 조사한다.

조사 항목:
- 최근 6개월 내 주요 뉴스 (서비스 출시, 실적, 전략 변화 등)
- 기술 블로그나 컨퍼런스 발표 내용
- 서비스 성장 지표 (MAU, 고객 수, 매출 등)
- 기업이 공개적으로 밝힌 기술 방향성

리서치 결과는 자소서 내에서 "관심을 가지게 된 계기"나 "지원 동기"를 구체화하는 데 활용한다.
단, 수집한 정보를 모두 나열하지 않는다. 자연스럽게 녹여내는 것이 핵심이다.

### 4단계: 초안 작성

각 자소서 항목별로 초안을 작성한다. 반드시 아래의 스타일 가이드와 글자 수 규칙을 따른다.

### 5단계: 피드백 및 수정

초안을 사용자에게 보여주고 피드백을 받는다. 톤 조정, 글자 수 증감, 강조점 변경 등
사용자의 요청에 맞게 수정한다.

## 스타일 가이드

자기소개서의 품질을 결정하는 핵심 규칙이다. 모든 항목 작성 시 반드시 따른다.

### 톤 & 어조: 사람이 쓴 것처럼

자소서는 AI가 아니라 지원자 본인이 쓴 글처럼 읽혀야 한다.

- **자연스러운 흐름**: "~했습니다", "~한 적이 있습니다", "~인 것 같습니다" 등 
  실제 사람이 쓰는 어미를 사용한다. "확신합니다", "매료되었습니다" 같은 
  과도하게 격식적인 표현은 피한다.
- **경험을 이야기하듯 풀어낸다**: "이전 회사에서 ~ 할 때", "~를 만난 적이 있는데" 같이
  대화체에 가까운 서술을 섞는다. 모든 문장이 같은 톤이면 기계적으로 느껴진다.
- **관심의 흐름을 보여준다**: 기업에 대한 관심은 "뉴스를 접하면서", "블로그를 읽다가",
  "서비스를 써보면서" 같은 자연스러운 계기로 표현한다.
  "귀사의 비전에 깊이 공감하며" 같은 형식적 표현은 사용하지 않는다.
- **구어체 표현을 적절히 섞는다**: "~도 있고요", "~한 편입니다", "~한 셈입니다" 같은
  표현을 2~3회 정도 넣어 딱딱함을 줄인다. 과용하지는 않는다.

### 금지 표현

다음 표현은 AI가 작성한 느낌을 주므로 사용하지 않는다:

- "확신합니다", "자부합니다", "매료되었습니다"
- "~에 부합합니다", "~와 일맥상통합니다"
- "핵심적인 역할을 수행하며", "차별화된 가치를 제공하겠습니다"
- "깊은 전문성을 바탕으로", "탁월한 역량을 갖추고 있습니다"
- 같은 구조의 문장을 3회 이상 반복 (예: "첫째, ~입니다. 둘째, ~입니다. 셋째, ~입니다.")

### 숫자와 성과 표현

숫자는 임팩트를 줄 수 있지만, 나열하면 보고서처럼 읽힌다.

- 한 문단에 구체적 수치는 최대 2개까지만 사용한다
- 나머지 성과는 "대규모 데이터를 다뤄본 경험", "쿼리 성능을 크게 개선한 경험" 같이
  정성적으로 표현한다
- 가장 인상적인 수치 1~2개만 골라서 사용하고, 나머지는 맥락으로 전달한다

### 구조

- 첫 문장에서 핵심 메시지를 전달한다. 두괄식이 기본이다.
- 단락 전환이 자연스러워야 한다. "또한", "더불어" 같은 접속사를 반복하지 않는다.
- 마지막 문단은 입사 후 포부나 다짐으로 마무리하되, 
  "최선을 다하겠습니다" 수준의 상투적 표현보다는 구체적인 목표를 언급한다.

### 항목별 작성 포인트

**지원동기 유형**:
- 기업에 관심을 갖게 된 구체적 계기(뉴스, 서비스 사용 경험, 기술 블로그 등)로 시작한다
- 기업의 방향성과 본인의 경험/가치관이 연결되는 지점을 보여준다
- 입사 후 하고 싶은 일을 구체적으로 언급한다

**직무적합성 유형**:
- 채용 공고의 필수/우대 역량에 직접 대응하는 경험을 중심으로 서술한다
- 문제 상황 → 본인의 접근 → 결과의 흐름(SCAR: Situation-Crisis-Action-Result)으로 풀어낸다
- 기술 키워드를 자연스럽게 녹여내되, 나열하지 않는다

**성장/포부 유형**:
- 과거 경험에서 배운 것과 앞으로 하고 싶은 것을 연결한다
- 해당 기업/팀에서만 가능한 성장을 구체적으로 언급한다

## 글자 수 규칙

자소서의 글자 수는 **공백을 포함한 글자 수(잡코리아 기준)**로 계산한다.

- 한글 1자 = 1자, 영문 1자 = 1자, 공백 1칸 = 1자, 줄바꿈 = 1자
- 쉼표, 마침표 등 특수문자도 각각 1자로 계산한다

지정된 글자 수의 95~100% 범위를 목표로 작성한다.
예를 들어 1000자 제한이면 950~1000자 사이로 맞춘다.

작성 후 반드시 다음 방법으로 글자 수를 검증한다:

```bash
# 공백 포함 글자 수 확인 (줄바꿈 제외)
cat <파일> | tr -d '\n' | wc -m
```

글자 수가 범위를 벗어나면 자동으로 조정한 뒤 결과를 사용자에게 알린다.

## 참고: 좋은 개발자 자소서의 원칙 (코드잇 스프린트 가이드 기반)

- 내가 하고 싶은 말이 아니라, 채용 담당자가 듣고 싶은 말을 쓴다
- 열정이나 열심을 단어로 강조하지 않고, 구체적인 경험과 결과로 보여준다
- "why"를 5번 반복하며 깊이를 더한다 (drill-down)
- 기업 정보를 탐색하고, 본인의 경험과의 연결 고리를 만든다
  (회사 홈페이지, 최근 3개월 뉴스, 기술 블로그, 채용 페이지 등 활용)
- 스토리텔링: 상황(Situation) → 위기(Crisis) → 행동(Action) → 결과(Result)
